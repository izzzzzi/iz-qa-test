# Блок 1: Тестирование требований — ЭТрН (Электронная транспортная накладная)

> **Примечание:** BPMN-диаграмма и XSD-схемы из задания находятся на Яндекс.Диске, который требует ручной капчи. Информация ниже основана на:
> - Документации [Контур.Диадок API](https://developer.kontur.ru/Docs/diadoc-api/docflows/waybill.html)
> - [Приказе ФНС № ЕД-7-26/1065@](https://www.nalog.gov.ru/rn77/about_fts/docs/11766120/) (формат ЭТрН)
> - [Постановлении Правительства №2200](https://normativ.kontur.ru/document?moduleId=1&documentId=441814) (правила заполнения)

---

## 1. Контекст бизнес-процесса

ЭТрН — цифровой аналог бумажной транспортной накладной. С **1 сентября 2026** — обязательна к применению (ФЗ №140-ФЗ).

### Участники процесса

| Участник | Роль | Тип подписи |
|----------|------|-------------|
| **Грузоотправитель** | Создаёт ЭТрН, подписывает T1 | КЭП |
| **Перевозчик** | Принимает груз, подписывает T2 | КЭП |
| **Водитель** | Подписывает приём/выдачу груза | ПЭП (простая) |
| **Грузополучатель** | Подписывает получение, T3 | КЭП |

### Структура документа (8 титулов)

| Титул | Описание | Обязательность |
|-------|----------|:--------------:|
| **T1** | Грузоотправитель — данные о грузе, маршруте, сторонах | ✅ Обязательный |
| **T2** | Перевозчик — подтверждение приёма груза | ✅ Обязательный |
| **T3** | Грузополучатель — подтверждение выгрузки | ✅ Обязательный |
| **T4** | Перевозчик — подтверждение выдачи груза | ✅ Обязательный |
| T5 | Изменение стоимости (перевозчик) | ⬜ При изменении |
| T6 | Согласование стоимости (грузоотправитель) | ⬜ При изменении |
| T7 | Переадресация | ⬜ При смене адреса |
| T8 | Эстафета (смена ТС/водителя) | ⬜ При смене |

### Основной сценарий (Happy Path)

```
Грузоотправитель → T1 (КЭП)
  → Водитель → T2 (ПЭП, замечания опционально)
    → Перевозчик → T2 (КЭП)
      → ГИС ЭПД (автоматически)
        → [Опционально: T7, T8]
          → Грузополучатель → T3 (КЭП)
            → Водитель → T4 (ПЭП, замечания опционально)
              → Перевозчик → T4 (КЭП)
                → [Опционально: T5, T6]
```

## 2. Чек-лист тестирования бизнес-процесса ЭТрН

### 2.1. Создание и валидация T1 (Грузоотправитель)

| № | Сценарий | Ожидаемый результат | Тип |
|---|----------|---------------------|-----|
| 1.1 | Заполнить все обязательные поля T1: грузоотправитель, грузополучатель, перевозчик, груз, маршрут, дата | T1 создан, валидация пройдена | Happy path |
| 1.2 | Подписать T1 квалифицированной электронной подписью (КЭП) | Подпись валидна, T1 переходит в статус «Подписан грузоотправителем» | Happy path |
| 1.3 | Попробовать подписать T1 ПЭП (простой подписью) | Система отклоняет: T1 требует КЭП | Негативный |
| 1.4 | Не заполнить обязательное поле (ИНН грузоотправителя, дата погрузки) | Ошибка валидации: «Поле обязательно» | Негативный |
| 1.5 | Указать несуществующий ИНН грузоотправителя | Ошибка валидации: «ИНН не соответствует формату» | Негативный |
| 1.6 | Указать дату погрузки раньше даты создания документа | Ошибка: дата погрузки не может быть раньше даты документа | Негативный |
| 1.7 | Заполнить T1 без грузополучателя (только при чьём-то участии) | Ошибка: грузополучатель обязателен | Негативный |
| 1.8 | Указать одинаковые ИНН грузоотправителя и грузополучателя | Ошибка: стороны должны различаться | Негативный |
| 1.9 | Ввести вес груза 0 кг | Ошибка: вес должен быть > 0 | Граничный |
| 1.10 | Ввести вес груза с десятичной дробью (>2 знаков после запятой) | Ошибка или округление до 2 знаков | Граничный |
| 1.11 | Создать T1, сохранить как черновик, не подписывать | Черновик сохранён, статус «Черновик» | UX |
| 1.12 | Проверить ID файла T1 по маске `ON_TRNACLGROT_...` | ID соответствует формату ФНС | Формат |

### 2.2. Приём груза (T2 — Водитель + Перевозчик)

| № | Сценарий | Ожидаемый результат | Тип |
|---|----------|---------------------|-----|
| 2.1 | Водитель подписывает T2 простой подписью (ПЭП) | T2 подписан, статус «Принят водителем» | Happy path |
| 2.2 | Водитель указывает замечания при приёме (повреждение упаковки) | Замечания сохранены, T2 подписан | Neg/Pos |
| 2.3 | Водитель отказывается подписывать T2 с указанием причины | T2 со статусом «Отказ», процесс приостановлен | Негативный |
| 2.4 | Попробовать подписать T2 КЭП на шаге водителя | Система ожидает ПЭП, КЭП не принимается (если роль — водитель) | Негативный |
| 2.5 | Перевозчик подписывает T2 КЭП после водителя | T2 полностью подписан, статус «Груз принят» | Happy path |
| 2.6 | Перевозчик подписывает T2 до водителя | Ошибка: порядок подписания нарушен | Негативный |
| 2.7 | Замечания водителя содержат некорректные символы (XSS) | Поле экранирует/отклоняет опасные символы | Негативный |
| 2.8 | T2 не подписан никем в течение N дней | Нотификация: «T2 ожидает подписания» | Таймаут |

### 2.3. Отправка в ГИС ЭПД

| № | Сценарий | Ожидаемый результат | Тип |
|---|----------|---------------------|-----|
| 3.1 | После подписания T2 КЭП — данные отправляются в ГИС ЭПД | Статус «Отправлено в ГИС ЭПД» | Happy path |
| 3.2 | ГИС ЭПД подтверждает получение | Статус «Зарегистрировано в ГИС ЭПД» | Happy path |
| 3.3 | ГИС ЭПД недоступна (сетевая ошибка) | Статус «Ошибка отправки в ГИС ЭПД», автоматический retry | Негативный |
| 3.4 | Ошибка валидации на стороне ГИС ЭПД (некорректный XML) | Статус «Ошибка ГИС ЭПД: некорректный формат» | Негативный |

### 2.4. Перевозка, переадресация (T7) и эстафета (T8)

| № | Сценарий | Ожидаемый результат | Тип |
|---|----------|---------------------|-----|
| 4.1 | Создать T7 при необходимости смены адреса выгрузки | T7 создан, статус «Переадресация» | Happy path |
| 4.2 | Указать в T7 дату/время и причину смены адреса | Поля валидны, T7 подписан | Happy path |
| 4.3 | Создать T8 при смене ТС или водителя | T8 создан, новые данные ТС/водителя зафиксированы | Happy path |
| 4.4 | Создать T7 без указания причины | Ошибка: причина обязательна | Негативный |
| 4.5 | Создать T8 с некорректным госномером ТС | Ошибка: формат госномера невалиден | Негативный |
| 4.6 | Создать несколько T7 подряд (цепочка переадресаций) | Каждая переадресация — отдельный T7 с уникальной датой | Happy path |
| 4.7 | T7 создана после выгрузки (T3 уже подписан) | Ошибка: T7 только до выгрузки | Негативный |

### 2.5. Выгрузка (T3 + T4)

| № | Сценарий | Ожидаемый результат | Тип |
|---|----------|---------------------|-----|
| 5.1 | Грузополучатель подписывает T3 КЭП | T3 подписан, статус «Груз получен» | Happy path |
| 5.2 | Грузополучатель указывает частичную приёмку (∑ количество < T1) | T3 с признаком частичной приёмки, система запускает возврат | Neg/Pos |
| 5.3 | Грузополучатель отказывается от приёмки (полный отказ) | T3 с признаком отказа, процесс возврата | Негативный |
| 5.4 | Водитель подписывает T4 ПЭП с замечаниями по выгрузке | T4 подписан, замечания сохранены | Happy path |
| 5.5 | Перевозчик подписывает T4 КЭП | T4 полностью подписан, процессинг завершён | Happy path |
| 5.6 | T3 подписан, но T4 не подписан — попытка завершить процесс | Ошибка: T4 обязателен | Негативный |
| 5.7 | Частичная приёмка — T7 возврата не создана | Ошибка: требуется T7 для возврата | Негативный |

### 2.6. Изменение стоимости (T5 + T6)

| № | Сценарий | Ожидаемый результат | Тип |
|---|----------|---------------------|-----|
| 6.1 | Перевозчик создаёт T5 с новой стоимостью | T5 создан, статус «Ожидает согласования» | Happy path |
| 6.2 | Грузоотправитель подписывает T6 (согласен с ценой) | T6 подписан, стоимость зафиксирована | Happy path |
| 6.3 | Грузоотправитель НЕ согласен с ценой — отказывает T6 | T6 со статусом «Отказ», перевозчик может скорректировать | Neg/Pos |
| 6.4 | T5 создана с отрицательной стоимостью | Ошибка: стоимость должна быть > 0 | Негативный |
| 6.5 | Создать T5 и T6 после завершения перевозки (нет активного процесса) | Ошибка: процесс уже завершён | Негативный |

## 4. Негативные проверки — детально

### 4.1. Нарушение последовательности операций

| № | Действие | Ожидаемый результат |
|---|----------|---------------------|
| N1 | Подписать T4 (выдача груза) до T3 (получение груза) | Система блокирует: T4 только после T3 |
| N2 | Создать T5/T6 (изменение стоимости) до завершения перевозки | Ок (можно на любом этапе после T2) |
| N3 | Создать T7 (переадресация) после подписания T3 | Блокировка: груз уже доставлен |
| N4 | Подписать T2 (перевозчик) до T2 (водитель) | Блокировка: нарушена последовательность |
| N5 | Попробовать подписать T3/T4 без T2 | Блокировка: T2 обязателен |

### 4.2. Некорректные данные

| № | Действие | Ожидаемый результат |
|---|----------|---------------------|
| N6 | ИНН перевозчика совпадает с ИНН грузоотправителя | Ошибка: одно лицо не может быть обеими сторонами |
| N7 | Адрес погрузки = адрес выгрузки | Ок: возможна перевозка на том же месте (например, между складами), но редкий кейс |
| N8 | Вес груза превышает грузоподъёмность ТС | Warning: предупреждение, не блокировка |
| N9 | Количество грузовых мест = 0 при указанном весе > 0 | Ошибка: если вес > 0, мест должно быть > 0 |
| N10 | Дата погрузки в прошлом относительно подписания T1 | Ошибка: дата не может быть раньше подписи |

### 4.3. Ошибки подписания

| № | Действие | Ожидаемый результат |
|---|----------|---------------------|
| N11 | Подпись T1 КЭП с истёкшим сертификатом | Ошибка: сертификат недействителен |
| N12 | Подпись T2 ПЭП — водитель не найден в системе | Ошибка: водитель не авторизован |
| N13 | Подпись T3 — подписант не сотрудник грузополучателя | Ошибка: нет прав на подписание |
| N14 | Попробовать подписать T1 ПЭП вместо КЭП | Ошибка: T1 требует КЭП |

## 5. Баг-репорты

### Баг-репорт №1: Некорректная обработка порядка подписания T2

| Поле | Значение |
|------|----------|
| **Заголовок** | T2 можно подписать КЭП перевозчика до подписания ПЭП водителя |
| **Приоритет** | Critical |
| **Окружение** | MVP ЭТрН, v1.0 |
| **Шаги** | 1. Создать T1 (грузоотправитель, подпись КЭП) 2. Открыть T2 3. Подписать T2 КЭП перевозчика 4. Вернуться к T2 |
| **Фактический результат** | T2 подписан КЭП, статус «Подписан перевозчиком». Водитель может подписать ПЭП позже. |
| **Ожидаемый результат** | Система блокирует подписание КЭП до подписания ПЭП водителем. Правильная последовательность: T2 водитель (ПЭП) → T2 перевозчик (КЭП). |
| **Severity** | Critical — нарушение бизнес-процесса: подпись перевозчика без фактического приёма груза водителем означает, что груз может быть «принят» без физического осмотра. |
| **Комментарий** | Согласно постановлению ПП РФ №2200, водитель первым подтверждает приём груза (с возможностью указать замечания). Перевозчик подтверждает после водителя. |

### Баг-репорт №2: Создание T7 (переадресация) после завершения перевозки

| Поле | Значение |
|------|----------|
| **Заголовок** | T7 (переадресация) можно создать после подписания T3 (выгрузка) |
| **Приоритет** | High |
| **Окружение** | MVP ЭТрН, v1.0 |
| **Шаги** | 1. Создать T1, T2 2. Подписать T3 (груз получен) 3. Попробовать создать T7 |
| **Фактический результат** | T7 создаётся, система позволяет указать новый адрес выгрузки |
| **Ожидаемый результат** | T7 недоступен после T3 — груз уже доставлен |
| **Severity** | High — данные перевозки становятся противоречивыми: груз «уже доставлен» по T3, но T7 указывает новый адрес |
| **Комментарий** | T7 должен быть доступен только между T2 (подписан) и T3 (подписан). |

### Баг-репорт №3: Валидация ИНН

| Поле | Значение |
|------|----------|
| **Заголовок** | Поле ИНН грузоотправителя принимает 11-значное значение |
| **Приоритет** | Medium |
| **Окружение** | MVP ЭТрН, v1.0 |
| **Шаги** | 1. Создать T1 2. В поле «ИНН грузоотправителя» ввести 11 цифр (например, 12345678901) 3. Подписать T1 |
| **Фактический результат** | T1 подписан, хотя ИНН должен быть 10 (юрлицо) или 12 (ИП) цифр |
| **Ожидаемый результат** | Ошибка: «ИНН должен содержать 10 или 12 цифр» |
| **Severity** | Medium — данные попадут в ГИС ЭПД с некорректным ИНН |
| **Комментарий** | Необходима валидация по маске: юрлицо = 10 цифр, ИП = 12 цифр, контрольная сумма |
